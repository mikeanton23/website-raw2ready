Developer Guide

Architecture • Development • Contribution • Extension

Introduction

Raw2Ready has been designed as a modular, extensible framework for industrial biotechnology data processing. Every major functionality is implemented as an independent component, allowing developers to extend the platform without modifying unrelated modules.

The project separates data processing, graphical user interfaces, configuration management and AI functionality into distinct layers. This architecture simplifies maintenance, improves testability and encourages collaborative development.

Whether you are implementing support for a new laboratory instrument, adding statistical analyses or developing AI agents, the framework provides clearly defined extension points.

Development Environment

A dedicated Python virtual environment is recommended for development to isolate dependencies from the system Python installation. 


git clone https://gitlab.com/arc3972240/bioindustry/raw2ready

cd raw2ready

python -m venv .venv

source .venv/bin/activate

pip install -r requirements.txt

After installing the dependencies, the graphical interface can be started with: 


python run_app.py

The command-line interface is available through: 


python raw2ready.py --help

Adding Support for New Equipment

New laboratory instruments can be integrated without modifying the overall workflow. Developers only need to implement a parser that converts vendor-specific exports into Raw2Ready's standardized internal data model. 


New Equipment

        │

Raw Export

        │

Equipment Parser

        │

Column Translation

        │

Standardized Dataset

Equipment Integration Steps

  1. Create a parser inside lib/parse.py.
  2. Register the parser in modules/parse.py.
  3. Define configuration parameters.
  4. Create a column translation template.
  5. Test using representative datasets.
  6. Update the documentation.

Coding Standards

Raw2Ready follows a consistent coding style to improve readability, maintainability and long-term collaboration.

Standard Recommendation
Python Version Python 3.11 or newer
Style Guide PEP 8
Formatting Black formatter
Imports Group standard, third-party and local imports
Documentation Docstrings for public functions and classes
Typing Use type hints where appropriate

Code Organization

        │

        │

        │

        │

User interface code should remain independent from data processing logic. Shared functionality should be placed inside reusable library modules.

Documentation Guidelines

  • Document every public function.
  • Include parameter descriptions.
  • Describe return values.
  • Add usage examples where appropriate.
  • Update documentation whenever functionality changes.
  • Keep examples synchronized with the latest release.

Git Branching Strategy

Development should follow a feature-based workflow to isolate changes and simplify code reviews. 


main

    │

    ├── feature/parser

    ├── feature/gui

    ├── feature/statistics

    ├── feature/agents

    └── bugfix/merge

Each feature branch should focus on a single functional improvement or bug fix.

Contributing

Contributions from researchers, software engineers and the industrial biotechnology community are welcome. Whether you are fixing bugs, adding equipment support, improving documentation or extending the AI framework, your contributions help improve Raw2Ready.

Before submitting a merge request, ensure that your code follows the project's coding standards, passes all available tests and includes any necessary documentation updates.
